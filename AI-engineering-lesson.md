# 消費工程部 AI 分享會 — 會議彙整報告

## 整體會議摘要

本次分享會由主講人帶領，對象為消費工程部工程師，主軸在於「如何在減輕工作負擔的前提下，用 AI 讓產出更好也更快」。主講人開宗明義提出核心假設：加速產出與減輕負擔兩件事並不衝突，而目前多數工程師「很認真、一直在做事，AI 卻沒有讓工作變輕鬆」的關鍵，出在**意圖（Intent）不清**與**理解成本過高**，而非工程能力不足。整場並非績效檢討，而是以工作坊形式先收集痛點、再逐一對應解方。

會議前段以兩個互動練習破題：先請每位成員憑印象估算「做事／等待／開會」的時間比例，接著收集每個人在工程工作上的痛點，並由主講人現場分類為「意圖問題、工作流問題、組織問題、模型（來源）問題」四類，得出「絕大多數痛點其實都是意圖問題」的結論。後段則依序講解三大工具與觀念：**產品意圖五問法**、**Mental Map／Mental Model（降低理解成本、以 Mermaid 圖視覺化）**，以及**AI 真正的加速來自非同步／多工與 Agent 分工（Planner／Builder／Test／Adversarial）加沙盒隔離**。

因時間不足，主講人將「系統如何建置」與 Evaluation Driven Development（EDD）的完整內容留待下週再講，並邀請成員先實驗本次教的做法、一週後回饋成效。全場多次以其開源專案（OpenAB）維護經驗與社群機器人 Jelly 作為案例佐證。（註：本次無投影片視覺資料，數據以逐字稿為準，凡涉及具體數值皆為口頭估算或工具名稱聽寫，已標註 (待確認)。）

## 一、工作坊目標與問題分層

主講人先建立整場的框架：把大家遇到的問題區分為「個人層次」與「組織層次」——個人層次的問題用工作坊當場解決，組織層次的問題則由主講人往上（跨部門）協助處理。主講人強調自己不是工程師，示範的做法若工程師採用「一定是我的十倍以上」，目的是驗證「減輕負擔」與「產出更快更好」可以並存的假設，並請成員誠實回饋哪裡痛、哪裡有問題。

1. 兩大主題：一是把個人／組織層次的痛點分流解決；二是示範「非工程師身份如何用 AI 加速產出」。
2. 定位澄清：本場非績效評比，主講人定位為「推動任務、陪大家一起想解法」的角色。
3. 觀察起點：工程師普遍很認真，但 AI 導入後主觀上並未變輕鬆，這正是本次想共同解答的問題。

## 二、工作時間分配盤點練習

第一個練習請每人憑印象（非精確計時）估算日常時間分配。原本分為「新功能／改需求／修 BUG／等待／開會」五項，後簡化為「做事／等待／開會」三項，讓成員報三個數字。

1. 分項最初為五項：新功能、改需求、修 BUG、等待、開會；後合併「新功能＋改需求＋修 BUG」為「做事」，簡化成做事／等待／開會三類。
2. 數值皆為主觀印象、成員自陳「隨便猜」，非投影片數據；各人所報三數字（如 6/2/2、1/2/7、2/1/8、1/3/6、5/1/4 等）僅供收集討論，精確值 (待確認)。
3. 討論延伸：多數人「等待」偏低；主講人指出「等待多」通常不是個人問題，而是組織環節未提前交付所致，屬組織面須解決的範疇。
4. 舊 AI 產出的問題被點名特別耗時，且「測試的時間比寫的時間多」。

## 三、工程痛點收集與分類

主講人現場畫表格收集每位成員的痛點，欄位含：是誰／痛點／上游或下游／迴圈斷在哪一站（釐清決策、驗收、迭代等階段）／個人解或組織解／時間。以下為收集到的主要痛點（依議程脈絡整理，負責釐清者為主講人與各當事工程師）。

1. 被隨時打斷、找的人太多，屬上游、組織面問題（會打斷開發專注度）。
2. 「說的跟做的不一樣」：已跟 AI 講清楚，但產出方向錯誤，屬驗收／意圖問題。
3. 迭代版本回歸問題：本來可運作的功能在某次迭代後失效，需判斷是否採用 AI 提出的解法，測試過程耗時。
4. AI 施作太慢：例如請 AI 重構 Codebase，預估一天的工作花了兩天仍未完成，屬意圖／規劃問題。
5. 決策點驗證慢：需快速驗證 AI 產出好壞以決定下一步，但每個決策點卡太久。
6. 複雜需求難以確保品質：需求簡單時直接把流程丟給 AI 即可，但需求複雜時 AI 易出錯、會自行揣測與規格不符的流程（此痛點由使用 Kiro 流程的成員提出）。
7. 新人上手困難：文件、資訊散落各地，需到處翻找（由到職近兩個月的成員提出）。
8. 最佳化建議無效：請 AI 幫忙最佳化，給了很多方向卻「一點用都沒有」；請 AI 加速改寫後跑起來也沒變快。
9. 測試太花時間：花較多時間在人工手測，期待能讓 AI 自行測試。
10. LLM 改版導致行為不一致（由使用 Skill、開發 Agent 的一位來賓提出）：Tool 端已很穩定，但底層 LLM Model 一改版，Agent 就像「失憶／人格改變」，忘記怎麼用既有 Tool、流程變奇怪；此屬模型來源端（非上述四階段）的問題。

**主講人的分類結論**：

11. 上述痛點多數為**意圖問題**（如「說的跟做的不一樣」「Rebuild／最佳化無效」「測試太久」），少數為**組織問題**（打斷多、頻繁開會）、**工作流問題**（需求需反覆 review），以及**模型來源問題**（LLM 改版行為漂移，屬無法迴避的外部變因）。
12. 核心判斷：如果大家很認真卻產出偏慢，「意圖」這一段一定有問題。

## 四、產品意圖與五問法

主講人主張：AI 做錯多半不是能力問題，而是「沒有被清楚告知該做什麼」，且往往有些「大家覺得理所當然、AI 卻不知道」的隱含資訊。主講人提出以「意圖卡／五問法」把意圖與邊界講清楚，並強調在工程上更關鍵的是 **intent border（邊界）**——不只講「要做什麼」，更要講「不要做什麼」。

1. 五問法五個問題：①這是誰的問題（誰 care）②做出來會改變什麼 ③成功的訊號是什麼 ④不做會怎樣（機會成本／Time window）⑤邊界是什麼（什麼不能壞、什麼刻意不做）。
2. 以「食安／驗毒油 App」為例走過五問：使用者非全體消費者、而是「在意吃到毒的人」；成功訊號是「有人在使用、有社群討論熱度」；不做的機會成本是錯失時間窗；邊界包含「產生速度不能太慢」「一定不做退款」等。
3. 核心觀點：Product Intent 是 AI 永遠做不好、且永遠需要人的部分，因為 AI 不懂外部世界的真實脈絡；以 Production 標準來看，模糊意圖產出的東西「永遠不夠好」。
4. OpenAB 維護實例佐證：主講人曾因未細看 PR 意圖，放行了一個「關掉刻意保留的 Bot-to-Bot Connection」功能的 PR 而被專案 Owner 糾正——AI 無法對 intent 做良好判斷，人必須自己講得清楚每個 review 的取捨。
5. 意圖的呈現方式可以是 PRD、可試作的 MVP、或一顆按鈕；主講人個人立場更激進：「要提需求的人都應先自己做一個 MVP，沒有 MVP 就不要來提」，因為現在模型光靠幾句話就能產出可討論的雛形。
6. 溝通應用：面對同事的需求，第一句話先肯定（三明治溝通），再請對方先把意圖用五問法討論清楚（可請對方「回去跟 AI 討論完再回來」），常能直接省掉一場會議；很多會議與反覆修改的根因，其實只是「彼此不懂對方在幹嘛」，而非真的意見衝突。

## 五、Figma 驅動開發的問題與意圖迭代

針對「拿到 Figma 就下去寫 code」的普遍場景，主講人指出 Figma 本質上像是一張影像檔（透過 Figma MCP 讀取，實際拿到的多是一包含 metadata 的 JSON，而非圖片本身），**不是好的產品第一意圖**；AI 的 OCR 能力差，Figma 轉 JSON 這一段常出問題。

1. 常見錯誤：介面元素位置跑掉、靜態圖無法表達不同場景條件下的互動與反應。
2. 建議中間層（由一位成員提議並獲認同）：在 Figma 到開發之間，補一個把「條件→情況→預期結果」寫出來的階段（GWT，Given-When-Then），讓 AI 產生並 review 是否符合畫面場景與流程，寫出的結果就不會差太多。
3. 主講人的處理流程示範：先用可把資料轉成 .md 的工具或 OCR 模型把 Figma 轉成扎實的 .md；再用五問法檢查產品邊界，若對方沒想清楚就把整包丟回去請對方先補齊。
4. **最重要觀念——迭代的對象是「意圖／Planning」而非「產品本身」**：
   - 錯誤行為（打地鼠式）：一開始只丟模糊需求，AI 產出後看到 bug 就一個個修，永遠在電腦前等它跑、改不完。
   - 推崇行為：先請 AI 出第一版 → 人看完後就每個要加的 feature（存法、顯示、是否與原架構相合）逐一「討論」→ 進入 Planning，把整個產品邏輯與初始準備全部釐清 → 確認後才進開發。這樣迭代次數大幅減少。
5. **Planning Review 是最累、最關鍵的一環（約佔專案主動出力的 80% (待確認)）**：一開始 Plan 錯，後面幾乎無法 recover，會一直混亂、改不完；若後面 testing 很痛苦或 build 兩天跑不完，代表 Planning 一定有問題。
6. 大型 Codebase（如兩萬行重構）的作法：回到 Context Engineering，在 Planning 階段建立 index／pseudo code，描述每個 section 該做什麼；並善用 Decoupling（解偶），若模組間 loosely coupling 甚至可平行處理、各自獨立開發後再合併。
7. 最終方法論：把 intent／planning 迭代到完全正確後，再走一次 **one-shot 流程**把它做完，是用 AI 開發產品最可能成功的方式。
8. 現場調查：已在用「規格先寫（SDD／Spec Driven Design）」做工程的有數人，回饋幫助蠻大；前端先前即有在做 SDD。

## 六、Mental Map／Mental Model：降低理解成本

主講人提出：在 Agentic Workflow 下，Bottleneck 已不是 code 產生，而是**人的理解**。主講人分享一套用視覺化圖降低理解成本的做法，並示範多個「魔法關鍵字」。

1. 魔法關鍵字：對 codebase 用 `explore`（比 `read` 更會到處探索、能讓較笨的模型變聰明）、`ELI5`（Explain Like I'm 5，用比喻把複雜東西講簡單）、`ASCII`（自動畫圖，但成品醜、不適合給人看）。
2. 推崇做法：用 **Mermaid Graph** 產生 Mental Map，push 上 GitHub／GitLab 利用其原生 Mermaid rendering 直接視覺化呈現，關鍵字為 `mental map`／`mental model`，通常 one-shot 即成、幾乎全對（主講人認為具備一定自主創意的模型更適合此任務）。
3. 效益：原本要跟 AI 反覆討論兩小時才懂的東西，視覺化後約 20–30 分鐘即可掌握其目的。
4. 實際應用：與合作夥伴把舊資料標籤改為新版本時，因「每天開會、沒人懂在幹嘛」，主講人把新架構與新舊差異全部視覺化（含比對表格），大幅減少溝通往返。
5. 看不懂時的逐級處理法：做 Mental Map → 仍看不懂就開 ELI5 版 → 仍看不懂代表 Gap 在更底層（某 Library 或 Concept），就請 AI 針對專案 Core Concept 另做一張解釋 Mental Map。心法：「看不懂不是因為我笨，是 AI 做得還不夠多」，把理解層的阻力全丟給 AI。
6. 極限與價值：對公司型產出仍會有解釋不出的怪問題——那就是「該拿去問人」的好問題；若問誰都不知道，恭喜你發現了一個值得改掉的地方。
7. 建議常規化：交付工具或資料給別人時，多花 5–10 分鐘做一張 Mental Map 一起交付，比 Document 更好懂（Document 常寫了到後面還是看不懂）。
8. GitHub／GitLab CLI（gh cli／glab）推薦用來處理版本控制等雜事（push／pull／commit／clone 等）。

## 七、AI 如何真正加速：非同步、多工與 Agent 分工

主講人拋出反直覺觀點：「AI 不是拿來加速單一任務的」——若你本來就是很強的 Coder，單一任務 AI 幾乎零加速（它處理可能花數分鐘，你手動只要幾十秒）。真正的加速來自**非同步處理與平行（多工）處理**。

1. 加速來源：把每個任務前段（約 80% 的意圖／規劃時間）處理好後交給 AI execute，人力就轉去做下一個任務；同一時間段能開越多平行處理，效率就翻越多倍。主講人表示目前可同時開約 15 個平行工作區 (待確認)，主要成本在每個任務「一開始」的處理。
2. Live Demo：用 GWS CLI（Google Workspace CLI）以語音輸入介面下指令，請 AI 自動建立會議邀請與會議連結，讓人可「事後不理、去做別的事」。
3. Context Switch 的解方：多工會忘記自己在幹嘛時，直接問 Agent「我現在在幹嘛」即可（也可用 recap／compact）。
4. **Agent 分工 Framework（Planner／Builder／Test／Adversarial）**：設一個只做 Planning 的 Planner Agent（大腦），下面有 Builder、Test，甚至接 CI/CD 的 code gate，全部跑完、Planner 覺得沒問題後才輪到人看。因 Planner context 不含 execution，較不會「跑到忘記在幹嘛」，也降低了需要 recap 的必要。
5. Adversarial Agent（對抗式審查）：要 AI 批評自己的產出時，不要讓「寫的那個 agent」自我 review（會合理化），而要找一個沒有 context 的 agent、給最少 context 去挑毛病；同理適用於人，不要讓提案者自己 review 自己的案。
6. Agent-first 介面：人只需把意圖告訴主 agent，後面全部跑；一個 Workspace 可含多個 Session，可用 sub-agent 分派高負載工作。搜尋則交給搜尋模型執行大規模搜尋。
7. **沙盒（Sandbox）隔離是安全關鍵**：Auto Mode 讓 AI 自動連續跑；Bypass 則完全不需人 Permission，風險極高。已有 AI 差點刪除整臺機器資料、以及 agent 直接刪除整個 Python repo 的案例。把每個 Workspace 關進獨立沙盒，出事只需砍掉那個沙盒、不會波及整臺電腦；Agent 在對齊誤差時可能被 social engineering，牆套隔離是可靠的規避方式。
8. 模型／Harness 輕重之分：Coding 介面本身就是 Harness，會改變模型行為，分 Harness Heavy 與 Harness Light；記憶體較重的模型不宜全開，主講人曾因同時開啟過多工作區導致電腦重開機，之後改用較輕量的 Agent 以便同時多開。

## 八、Evaluation Driven Development（EDD）與交付門檻

回應「最佳化跑太多輪、很累」的痛點，主講人提出從 TDD／SDD 進一步轉向 **Evaluation Driven Development（EDD）**。

1. 概念：不只丟 spec，而是把「該怎麼做、不該怎麼做」用自然語言或 code 寫成一整套評估（Evaluation，比喻為「考古題」）。
2. 流程：Adversarial Agent 依這套 Evaluation 反覆執行、驗收，AI 必須「考到 100 分」才輪到人看；任一題錯就 feedback 回 Planning、再叫 Execute 改，來回數百遍後才交到人手上。
3. 效益：依分享經驗，人需要看的 iteration loop 可從約 30 圈降到約 3 圈；成效 100% 取決於 Evaluation 怎麼寫，本身就是一個工程過程。
4. 定位：整場談的都是「後設」層——不是你實際怎麼做工作，而是你怎麼建立一套系統（Evaluation／關卡）來替你做工作。
5. 針對「要 AI 提出想不到的最佳化方向」：先給方向感（明確指出目前做法哪裡不滿意），再叫 agent 上網查（web search）別人怎麼做、別人的 priority，帶回 feedback 後再重新檢視。
6. 給同事需求時的可協助方式（來賓分享通用團隊經驗）：把想看到的 template 先丟到需求方，請對方先填好、share 到 Google Drive，會議先讀完再開；對方開會從一小時壓到多半 15 分鐘內結束。搭配「先各自讀完同一份 PRD／文件再開會」的方式，主講人表示會從自己主持的會議開始推行。

## 九、OpenAB 開源專案與社群機器人 Jelly 分享

主講人以其參與的開源專案 OpenAB 收尾，展示可套用上述方法的成熟案例（因時間不足多為預告，完整內容留待下週）。

1. OpenAB 已有相對成熟、社群 based 的 review process，整個 maintain 幾乎由一人在跑，主講人只需發 PR；其文件特色是「讓 agent 直接讀就知道怎麼做」。
2. 心智模型比喻：可把日常工作理解為「維護一個超大的 Public Repo，每個人交的每件事都是一個 PR」——PR intent 不好就先修 intent，修好再做。
3. 社群機器人 Jelly（現場 demo）：OpenAB 社群的友善小水母機器人，被 @ 時以會跑動的 emoji 呈現 Loading／思考中的「情緒價值」；可由專案維護者授權執行特權工作、協助社群問題排查。
4. 主講人坦言起初連 worktree 是什麼都不知道，強調工程師的好處是「需要時再學」。
