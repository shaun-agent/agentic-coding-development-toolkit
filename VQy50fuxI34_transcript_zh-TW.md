# 別再談 Loop Engineering：打造 AI 開發工作流程

來源：[YouTube 影片 VQy50fuxI34](https://www.youtube.com/watch?v=VQy50fuxI34)。以下是依原始英文逐字稿整理的繁體中文翻譯；為了閱讀順暢，將零碎語句合併成段，時間碼仍對應原影片。

## 別再把完整工作流程縮減成「迴圈」

**[00:00:00–00:01:20]**

是時候談談所謂的「Loop Engineering（迴圈工程）」了。我原本希望這個詞流行幾週之後就會自然消失，也希望有哪位頂尖工程師會直接指出這個說法有多荒謬，但沒有人這麼做，所以只好由我來說。

忘掉 Loop Engineering 吧。若要穩定而大規模地使用 Agent 建造有價值的軟體，這是錯誤的思考方式。Loop Engineering 只是把軟體開發生命週期重新包裝了一次，而且這個名稱既不清楚，又充滿炒作意味。這支影片要把它簡化，還原成它真正的樣子。只要正確理解接下來要拆解的概念，你就能大幅領先目前的 AI 產業。資訊愈清楚、愈簡單，工作就愈快、表現也愈好。

更有價值的理解方式，是把使用 Agent 開發軟體，想成在一座「軟體工廠」裡打造開發工作流程：提示進入工廠，特定工作流程開始運作；每一套工作流程都是「程式碼加上 Agent」的組合，最後產出結果。別再把時間花在 Loop Engineering 這個名稱上，把珍貴的工程時間與 Token 用來打造 **AI 開發工作流程（AI developer workflows）**。

## 講者背景，以及為什麼要提出反對意見

**[00:01:25–00:03:37]**

先說，我憑什麼挑戰 Anthropic 的 Boris Cherny、現在任職 OpenAI 的 Peter Steinberger 等 AI 工程師所提出的重要觀念？如果你已經知道我是誰、做過什麼，或根本不在乎，可以直接跳到畫面上的時間點。

我是 Dan Isler，也就是 IndieDevDan。我有超過十五年的軟體工程經驗，最早和兄弟們用 ActionScript 2、ActionScript 3 製作 Adobe Flash 遊戲；曾在 Blizzard 實習，後來很快進入金融與會計領域，用過 C#、TypeScript、Python、糟透了的 React——也感謝尤雨溪創造 Vue。

在還沒有人替「AI 寫程式」取名字以前，我就已經在使用 Aider，以及 GPT-3.5 Turbo、GPT-4、Sonnet 3 這些現在看來很經典、但你可能從未聽過或使用過的工具與模型。我擁有 agenticengineer.com，數千位來自知名公司的工程師，透過我親手製作的資訊產品改善了自己的 Agentic Engineering 能力。

當然，任何人都能買網域、開課程。但我也在 GitHub 與這個 YouTube 頻道上，留下了無可否認的程式碼與內容紀錄；多年來每週持續發布。你可以看到我和追蹤這個頻道的工程師，長期走在 AI 產業曲線之前。我不像其他科技內容創作者，只是追逐新聞來換取觀看數；在這個頻道，我們思考、規劃，也真的動手建造。

每隔幾個月，我都必須重新說明一次：我不是內容創作者，而是一位把內容創作當副業的軟體工程師。為什麼？因為這項技術太有價值，不能只落在少數幸運的人手裡。沒錯，我賣課，也會因此受益，那又怎樣？不要把自己最擅長的事永遠免費送人；歡迎來到資本主義。

關於我的部分到此為止。我不是每個星期一出現來自誇，而是要給你這樣的工程師一項優勢，讓你在 AI 時代加速自己的職涯、工作、事業與工程能力。不談炒作。

## 價值創造的三個角色：工程師、Agent、程式碼

**[00:03:37–00:04:46]**

忘掉 Loop Engineering，改把注意力放在這件事上：現在的工程工作，有三種創造價值的角色——你我這樣的工程師、Agent，以及程式碼。Agentic Engineering（代理式工程）的核心，就是知道應該在何時、何處使用哪一種角色。

你可能會想：這和 Loop Engineering、開發工作流程有什麼關係？請先跟上來，我們會逐步推導到那裡。掌握基本原理，才能掌握更複雜的組合。

在這場 AI 迷狂裡，大家似乎忘了：程式碼很快，除非你刻意改變它，否則每次都會以相同方式執行；而且它不花 Token。真正以光速運作的東西，就是程式碼。採用這三種價值角色中的任何一種，都有隱含成本。大家都在談 Agent，也很清楚工程師的成本，然而在穩定創造價值這件事上，程式碼才是沒有得到足夠肯定的英雄。穩定的價值創造，才能帶來穩定的商業價值。

若比較可靠度，程式碼遙遙領先，其次是工程師，最後才是 Agent。接下來，就從最基本的開發工作流程開始。

## 從最小流程，逐步加入確定性驗證

**[00:04:50–00:05:20]**

最簡單的基礎流程是：工程師向 LLM 下提示，接著由工程師審查結果。往後每一個迴圈、每一套工作流程、每一項工作，都建立在這個基礎之上。

當然，現在我們不再只使用裸 LLM；中央節點已經換成 Agent。你可以放入自己偏好的 Agent 或模型，現在模型本身已經不是重點，重點是你我每天實際執行的工作流程。接著，我們把它放大。

**[00:05:20–00:06:50]**

現在，程式碼、Agent 與工程師都加入流程，我們開始建立愈來愈進階的開發工作流程。這裡有一個很重要的變化：我們加入了程式碼。最初只是執行 Linter，並設置一個條件。如果 Linter 失敗，結果就送回建置 Agent（Build Agent）；如果成功，就繼續通過。這個條件與回到建置 Agent 的路由，形成第一個迴圈，「Loop Engineering」一詞也由此而來。

但 Loop Engineering 太簡化，也不精確，因為整件事遠不只這個迴圈。下一步要如何持續改善工作流程、得到更好的結果？當然是加入更多具確定性的程式碼。

現在可以放入多個「通過／失敗」條件，把失敗結果送回你的建置 Agent——不論你用的是 Codex、Claude Code、Pi Coding Agent，或任何 Agent Harness。這才是現在使用 Agent 建造系統的基礎：工程師、Agent 與原始程式碼三者都在其中，你必須在正確時間、正確位置運用它們。

例如再加入格式化工具。無論使用什麼語言，都可以執行 Lint、格式化與型別檢查，然後繼續擴充驗證迴圈，把失敗資訊送回 Agent。這些條件確實形成了所謂的「迴圈」，但真正發生的事，是資訊在整個系統工作流程中的移動。

**[00:06:50–00:07:35]**

繼續擴充。下一項很有價值的程式碼，是測試。把 Lint、格式化、型別檢查和測試的結果，一次又一次送回建置 Agent，直到全部通過，最後才交給工程師審查。

你會看到一個規律：你我總是出現在流程兩端。這正是 Agentic Engineering 的兩個限制點——前端是提示，也就是規劃；末端是審查，也就是驗證。若你正確地大規模實施 Agentic Engineering，除了少數例外，人應該出現在開頭和結尾。AI 開發工作流程可以從這種簡單形態開始，但應該繼續成長，因為真實的工程工作遠比這複雜。

## 把驗證與規劃交給專門 Agent

**[00:07:40–00:08:23]**

我們可以把測試、驗證、Lint、型別檢查等工作，集中交給一個測試 Agent（Test Agent）。如此一來，我們是用更多運算資源擴大影響，也用更多運算來提高信心。

想像我們已把所有想做的檢查交給測試 Agent：如果出錯，就把完整脈絡送回建置 Agent；如果通過，才由工程師審查，接著交付成果、發布程式碼。

這裡有幾個反覆出現的主題：工作流程擴大時，你可以增加 Agent，也可以增加程式碼；但除了「打造那個會打造系統的系統」之外，不應該增加更多工程師的操作負擔。

**[00:08:23–00:09:08]**

再把規劃加入工作流程。這些觀念其實都很熟悉：以前，工程師自己規劃、建置、測試，再請另一位工程師審查，最後部署到正式環境。這本來就是一套開發工作流程；現在所做的，只是把 AI 加進去。

迴圈只是其中一小部分。你當然可以稱它為 Loop Engineering，但這個詞並未涵蓋全貌。如果有「迴圈工程」，那條件判斷是不是也要叫「條件工程」？函式要叫「函式工程」？軟體開發生命週期中每一種控制流程，是否都要在名字後面加上 Engineering？這樣永遠說不完。

## 工作樹帶來平行性，沙盒帶來完整隔離

**[00:09:08–00:10:28]**

目前很常見的模式，是把每個 Agent 放進自己的工作樹（worktree）。這能提供隔離與平行性，讓更多工作同時進行，還有一個很好的副作用：Agent 不會彼此踩到對方的修改。

我們因此會再寫一段提示，只是這次把提示寫進程式碼，由程式碼建立工作樹。這段具確定性的程式碼會根據提示啟動多個工作樹，並分別執行多個 Agent。換言之，我們再次用運算資源擴大影響。

多個工作樹可以同時執行由規劃 Agent（Planner Agent）、建置 Agent、測試 Agent、審查 Agent（Review Agent）接手的「規劃、建置、測試、審查、合併、發布」管線。這正是過去工程師親手走過的開發工作流程；如今加上 AI，所以稱為 AI 開發工作流程。

你應該把工程時間集中在這個問題上：我要如何組合工程師、Agent 和程式碼這三種價值創造角色，打造能替我、替公司、替使用者與產品執行大量工作的流程？這才是勝負所在。

**[00:10:28–00:11:58]**

工作樹是很好的起點，但不是理想的終點，因為它仍有許多問題。更進一步，可以讓每個 Agent 都擁有自己的沙盒。換句話說，不只是啟動工作樹，而是讓每個 Agent 擁有一部自己的電腦，也就是獨立運作的 Agent 沙盒。

如此便能得到完整隔離。你也可以親自進入沙盒，檢視工作內容、結果、網頁、測試或應用程式，完成任何必要審查。所有工作回來之後，再統一合併並發布。

再次注意，三種價值創造角色正在一起運作。你能設計多少這樣的 AI 開發工作流程，就能用 Agent 把自己的影響力擴大多少。對工程師而言，設計這些流程正是最能創造價值的工作。所以我總是在頻道上說：**要打造那個會打造系統的系統。**

你會逐漸發現，這些流程和你看過、做過的許多工作很相似；也希望你正在把它們導入團隊、同事、公司與工具之中。不過這才剛開始。真實工作會愈來愈複雜，而這正是 Agentic Engineering 的藝術與科學。

## 從 Kanban 工單進入完整交付管線

**[00:12:02–00:13:03]**

隨著流程成熟，常見的下一步是建立 Kanban 看板或某種工單系統。輸入會從組織各處湧入：客服、產品，也可能是工程師。此時，我們在程式碼外又多了一個新的工作單位與包裝層，也就是工單系統。

在 Tactical Agentic Coding 中，我會做一個非常重要的區分：當下真正需要投入心力的，是包覆應用程式的 **Agentic Layer（代理層）**，包括 Agent、提示、Skill 與系統提示。當你把這一層和程式碼、系統、整個團隊及組織組合在一起，才真正是在做 Agentic Engineering。

先說結論：Agentic Engineering 不只是 Agent。它也關乎你的團隊，更重要的是你的使用者；你要把最有價值的角色組合成一套完整技術棧。

**[00:13:03–00:15:16]**

對多數團隊來說，工單會先由真正了解現況的工程師分析，再翻譯成中階或低階提示，交給另一套完整工作流程或 Agent 沙盒。但進階團隊若能教會全組織寫出夠好的提示，並隨著模型能力提升，就可能跳過工程師改寫提示的步驟。工程師的工作應該是打造系統；我們要在能對整個組織產生複利的後設層工作。只要前一步做得夠好，進階團隊就能省略這道工程師提示。

Kanban 看板本身只是程式碼，並沒有 Agent。接下來進入流程核心：程式碼先把工單移到「規劃中」，然後 Agent 接管管線。

偵察 Agent（Scout Agent）會尋找相關程式碼、工單、文件與既有規格檔，再把資料交給規劃 Agent。也就是把搜尋與規劃拆分給兩個 Agent，再次用運算資源擴大影響。規劃完成後，程式碼更新工單、搬移脈絡，並在沙盒內完成特定準備工作。

接著啟動建置 Agent。你很熟悉後面的流程，因為自己做過無數次：建置 Agent 完成後把任務移到測試；測試 Agent 執行測試。這個單獨迴圈只是整套工作流程的一部分，它會持續執行，直到結果通過。然後運行 CI/CD；CI/CD 也可能通過，或失敗後回到建置 Agent 修正。最後離開沙盒，由工程師審查程式碼：失敗就退回，通過就發布。

因此，整件事遠超過 Prompt Engineering，也超過 Context Engineering、Harness Engineering 或 Loop Engineering。它描述的是一個團隊如何帶著內部所有角色，像一個有機體般運作。即使你只是單人開發者，道理也一樣：你、Agent 與程式碼如何共同產生有價值的成果。

## 正式環境故障：專用 Hotfix 工作流程

**[00:15:16–00:15:46]**

但這仍不是終點，工程的未來非常廣闊。想像另一個場景：客服面臨危機，正式環境離線、系統已經崩潰。

此刻，你的組織事先準備了什麼 AI 開發工作流程？正式環境故障時，你要如何運用工程師、Agent 與程式碼解決問題，阻止公司因系統停擺而持續損失金錢？

**[00:15:46–00:16:32]**

讓我們走一次流程，因為這個情境應該事先想過，也應該預先設計和架構出對應的 AI 開發工作流程。

客服建立工單；在這個例子中，訊息直接送到 Slack、Teams 或公司的溝通頻道，由一位頂尖工程師立即接手。工程師向偵察 Agent 下提示，再由它把任務路由到熱修復 Agent（Hotfix Agent）。

熱修復 Agent 擁有一套專門的記憶與知識，是以「盡快修好」為最高優先的專家。它不是來追求最標準、最漂亮的作法，也不負責最佳化任何東西；唯一任務，就是盡快送出修復。它是一個做外科手術式修補的客製 Agent，把你們的工程經驗模板化在其中。

**[00:16:32–00:17:43]**

接著要讓人加入迴圈。這是正式環境熱修復，我們必須確認解法真的有效，因此需要工程師投入判斷，選擇核准或拒絕。這形成一個迴圈。

一旦核准，就建立多個沙盒，讓同一解法平行執行。我要的是第一個找出可行解法、速度最快的 Agent 勝出。你可以依運算預算放大或縮小：對複雜的正式系統，可能同時讓三個、五個甚至十個 Agent，在各自沙盒中競速尋找解法。

你不用介意它們彼此競爭，因為已經準備好運算能力，也已透過 Agentic Engineering 擴大影響。每個 Agent 都在自己的沙盒中迭代，直到產生通過結果；失敗分支就回到熱修復 Agent 與工程師處理。一旦其中一個沙盒率先找到可行修復，工程師立即完成最終驗證並發布。

問題是：你的組織是否已經有一套處理正式環境崩潰的 Agentic 工作流程？你能否運用 Agent 時代的三種價值角色——工程師、Agent 與程式碼——以破紀錄的速度恢復系統？三者缺一不可。

## 多種專用流程，組成軟體工廠

**[00:17:43–00:19:11]**

這套方法可以持續擴大，最後形成真正的軟體工廠。工廠中有許多不同類型的專用 Agent 沙盒工作流程：有的處理例行雜務，有的處理 Bug，有的開發功能，也有剛才走過的熱修復流程。

概念很清楚：任何需要的專用 AI 開發工作流程，都能先建好，再由路由系統把任務送到正確位置。這就是 Agentic Engineering 的藝術與科學，是所有元素組合起來的完整樣貌；迴圈只是整張圖裡很小的一部分。

其中還包含大量 Prompt、Context 與 Harness Engineering。實作方法可以有上百萬種，多 Agent 協作模式也有無數選擇。關鍵只有一件事：在正確時間使用正確組合，讓工程工作透過 Agent、程式碼與工程師，從頭到尾流動並完成。

我知道自己一直重複，這是刻意的。任何領域的成功，通常都來自持續做、持續說、持續專注在少數幾件真正重要的事。

## 一座完整軟體工廠如何運作

**[00:19:11–00:20:00]**

現在走一次完整的軟體工廠，仍以 Kanban 工單為例。任何人都能提出工單，內容可能是功能、Bug 或例行工作。

進階團隊不會浪費工程師時間，把每張 Kanban 工單重新翻譯成低階或中階工程拆解；工單一進來，就直接啟動軟體工廠。工廠開始後，先把工單標成「進行中」，接著交給工廠路由 Agent。這個節點可以只是一次簡單的 LLM 呼叫，也可以是一段確定性程式碼；具體如何選擇由你決定。

這裡先放一個工廠 Agent 接收結果，快速查看程式碼庫，判斷這個系統應該執行哪一套 AI 開發工作流程。

**[00:20:00–00:21:19]**

但第一步仍是建立沙盒。我們不再限制 Agent 的運作空間，也知道 Agent 的用量只會繼續增加。這正是運算資源短缺的一部分原因。除了強化學習擴展與其他機器學習工程需求外，我敢保證，Agent 沙盒終將占據全球多數電腦運算。你我親自使用的裝置會愈來愈少，而 Agent 使用的沙盒會持續增加。

沙盒準備完成後，Agent 已判斷應採用哪種流程，才能以最佳成本、最佳效能和正確速度完成任務。你不會拿處理熱修復或大型功能的重型流程，去做一件簡單雜務。功能開發時，建置 Agent 或許可使用高性價比的主力模型，而規劃與偵察 Agent 使用最先進模型，以免遺漏重要資訊。這裡當然還能加入一整套多 Agent 協作機制。

重點是：不要為例行雜務部署重型 AI 開發工作流程。簡單雜務只要交給一個使用主力模型、甚至輕量模型的 Agent，完成建置、執行 Lint 與 CI/CD，再由工程師審查、發布即可。

稍後會談 ZTE（Zero-Touch Engineering，零接觸工程）。不過最成熟的團隊將逐漸拿掉工程師審查，因為他們已建立出足夠可靠、確定能替自己執行工作的系統。但每套獨特流程之所以不同，都有其理由；你應該打造多套 AI 開發工作流程，而不是只有一套。

**[00:21:19–00:22:48]**

稍後我會提供一些建立流程時非常實用的建議。一般原則是從簡單開始；逐步擴充後，最終會得到一座能操作應用程式的軟體工廠。如果做得正確，它甚至會比你和工程團隊親自操作得更好。

這也說明了為什麼現在所有工程心力都應投入 Agentic Layer，而不是 App Layer（應用層）。應用層交給 Agent；最優秀的工程團隊幾乎不再親手修改產品。我知道這個說法可能很有爭議，有些工程師不喜歡聽，但頂尖團隊是在 Agentic Layer 進行後設工程：打造那個會打造系統的系統。這是 Tactical Agentic Coding 的核心論點。

數千名工程師已經理解這件事，你遲早也會理解。這不代表工程師永遠不能進入應用程式工作；但當產品成功、擁有大量使用者後，真正的重點是建立一座軟體工廠，讓它比工程師單獨作業、程式碼單獨運作或 Agent 單獨工作都更好。仍然是那三種價值創造角色：Agent、工程師、程式碼。

## 目標不是「迴圈」，而是可重複的工作流程

**[00:22:48–00:23:33]**

這一切導向一個簡單結論：你真正需要的不是迴圈，而是 AI 開發工作流程。

你可能會說：「但你剛剛明明畫了一大堆迴圈，這不就是迴圈嗎？」好吧，如果你想叫它迴圈，我也不在意；只是「迴圈」這個概念限制太多。照這個邏輯，我們還需要 If Engineering、Throw Engineering、Exception Engineering，所有東西都得取一個 Engineering 的名字。

這其實就是開發工作流程。以前由工程師判斷一項工作是例行雜務還是 Bug，撰寫計畫、執行計畫，依此類推。如今只是多了一種工具——Agent——所以我們現在擁有 AI 開發工作流程。

**[00:23:33–00:25:04]**

在最高層次的 Agentic Engineering 中，你要打造的是軟體工廠：在整個組織中，以正確的工程師、Agent 與程式碼組合，執行正確的工作。規模繼續擴大後，還會加入其他團隊成員和橫跨公司的各種職能。但核心上，工程師仍對程式碼負責。

許多組織在納入其他成員時會遇到問題，尤其是那些怎樣都寫不清楚工單的人。你一定經歷過：產品經理、CTO 或技術主管丟出一張糟透的工單，最後還是工程師負責翻譯。這裡還有許多關於人與組織層級的工作要處理。

不過歸根究柢，仍然是工程師加上 Agent、再加上程式碼，共同構成 AI 開發工作流程。價值將在這裡以驚人的程度與規模產生。

一旦做對，並設置正確的護欄與 Harness——包括 Prompt、Context、Harness Engineering 等所有部分——你就擁有一套可重複執行的工作流程。只要把自己的工程專業模板化，寫入 AI 開發工作流程的結構裡，它便能執行數十、數百甚至數千次，一再交付穩定結果。

## 專業化，才是最大的槓桿

**[00:25:04–00:26:24]**

長期以來，我一直反對只使用開箱即用的通用 Agent。專業化才是勝負關鍵。

什麼是產品？什麼是公司？除非你是大型科技巨頭，否則一項產品或一家公司的本質，就是一群人和一套技術，替特定使用者輪廓、特定使用者或特定客戶，解決特定問題。依定義而言，這就是專業化。

現在最有價值的資產，是你的專業知識。你可以把它模板化到工程方法中，再寫入 AI 開發工作流程。這正是 Agentic Coding 最大的槓桿點：建立完整的 AI 開發工作流程，把所有元素組合起來。

所以，我們再次遠離 Vibe Coding。這不是 Vibe Coding。Vibe Coding 是不知道系統如何運作，也不去查看系統如何運作；Agentic Engineering 則是因為太清楚系統會如何運作，所以不需要每次親自查看。

工程師已向上移動一層，開始做後設工程：在正確時間，以正確效能、價格與速度，把工程師、程式碼與 Agent 三種角色最佳化成正確的開發工作流程，藉此累積一項會產生複利的優勢。

**[00:26:24–00:26:49]**

時間一久，你會意識到另一件重要的事：這些開發工作流程也會被放進產品，直接服務客戶，讓客戶本身成為流程節點。同樣地，在公司裡，每個能向系統下提示、並從系統取得結果的使用者，也都是節點。

這套系統必須由你設計，但它本質上仍只是另一套系統。

## 三項實作建議

### 一、從最簡單的流程開始

**[00:26:49–00:27:55]**

我大概已經寫過數百、甚至數千套 AI 開發工作流程。以下濃縮我目前所有經驗，以及給工程師的建議。

第一，保持簡單。開始建造時，先選你想得到的最簡單工作流程。一般人啟動 Agent 後，會來回下提示、守在旁邊照看；大家都知道這是什麼樣子。第一步只要讓它自動對程式碼執行 Lint。

為了讓差異真正清楚，請把步驟拆開。我不是叫你寫一個 Skill，讓 Agent 建置完後，在 Skill 最底部順便跑 Lint；而是使用 Agent SDK 啟動建置 Agent 執行工作，接著由獨立程式碼執行 Linter。如果 Linter 失敗，就用同一個 Session ID 把結果送回建置 Agent。

你必須把程式碼和 Agent 分離。否則，你得到的只是「一個 Agent 在呼叫程式碼」，並不是我們想要的架構。我們要在整條流程中維持關注點分離。如果現在還聽不懂也沒關係，實際開始建造後就會明白。

不要把一百個工作流程節點全部塞進一個巨大 Skill，這會造成嚴重的測試與驗證問題。

**[00:27:55–00:29:02]**

接著逐步增加幾個節點，開始解決真實問題。執行型別檢查和 Linter；任何一步失敗，就把資訊匯回建置 Agent。此時你會看到，一個能自行運作、規模更大的系統正在成形。

人只在開頭和結尾出現，也就是 Agentic Coding 的兩個限制點：規劃與審查；中間其餘工作由系統完成。

發展到一定程度後，再把不同 Agent 拆開並專業化。例如拆分前端與後端，或拆成建置與測試。關鍵仍是將 Context 拆分清楚，讓脈絡能在各個 Agent 和程式碼之間移動。

剛開始時記住 KISS：Keep It Simple, Stupid（保持簡單）。你完全可以先採用純 Skill 型工作流程，除了開頭提示和結尾審查外，全部放在單一 Skill 裡。但一旦要進入正式環境、認真經營 AI 開發工作流程，就必須把程式碼從 Skill 中拆出去；否則那些程式碼仍只是由 Agent 內部呼叫。

每個步驟都必須非常清楚，才能在 AI 開發工作流程中設置正確護欄與資訊流。

### 二、先親自走完流程，再把它系統化

**[00:29:02–00:30:08]**

第二項重要建議：設計 AI 開發工作流程以前，先親自做過一次完整工作。很多工程師聽了會覺得非常痛苦，但你仍可以在終端機使用 Agent，跑建置、執行測試。我不是要你徒手完成，那在今天只是浪費時間。

我的意思是，不論打算建立什麼工作流程，都先從頭到尾跑一次：親自走進每個節點，執行每條路徑與條件，觀察函式如何運作，完成審查，再發布到正式環境。之後，才開始把這一切改寫成工程師、Agent 與程式碼的組合。

我建議用 Mermaid 一類的工具把流程畫出來。順帶一提，影片裡的動畫就是改編自 Mermaid 圖：我讓 Agent 使用規劃、建置、測試流程，一次完成開發工作流程，做出以 Mermaid 圖為基礎的動畫應用程式。向 Mermaid 和 mermaid.live 致意。

因此，第二項建議就是先親自走過全部流程。拿紙筆坐下來，或使用 Mermaid 等工具，認真把整套工作流程寫清楚。

### 三、同時使用 Agent 與程式碼

**[00:30:08–00:31:24]**

最後，別只使用 Agent，要同時使用 Agent 和程式碼。起步時可以先用 Agent 加 Skill；但一進入正式環境、開始認真處理這套系統，就應把部分 Skill 工作移到程式碼。

原因不只是 Token 成本，更包括效能、可靠性與速度。在這場 AI 迷狂中，大家似乎忘了：程式碼的速度不花 Token、不會產生幻覺，每次都做完全相同的事，而且真的以光速運作。因此不要過度依賴 Agent，要用真正的程式碼執行來平衡。

當然，其中仍需要資訊協作，而這正是 Context Engineering。每個步驟之間，都要有存放結果的位置。這會花時間，也可能很煩；過程中你會想：「乾脆全部塞進同一個 Skill 算了。」但長期來看，那會是錯的，我可以保證，因為我已經走過那條路。不要重複其他工程師已犯過的錯；隨著規模增長，把它們拆開。

所以第三項重點是：使用 Agent，也使用程式碼。兩者結合勝過任何一方單獨運作，尤其當流程成長為真正大型、能替你與組織執行嚴肅工作的 AI 開發工作流程時。

**[00:31:24–00:31:58]**

為什麼？因為你必須測試每個節點：測試規劃、建置，還要測試從規劃到建置、更新狀態、進入測試及失敗回傳的整條路徑。它仍是一套系統，而你這位工程師要對它負責。

所以請繼續使用優良、經典的工程模式：可隔離、低耦合、單一介面。這些原則現在可能比以前更加重要。一旦正確設置 AI 開發工作流程，它就會被重複執行數百、數千次；Agent 加上程式碼，也會將結果成倍放大。
